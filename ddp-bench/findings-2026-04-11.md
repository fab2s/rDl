## Key Discovery: Solo Baselines Are Unreliable

MLP solo-0 diverges after epoch 3 (loss: 0.004 -> 0.742 -> 1.278 -> ... -> 0.511).

**Before any DDP analysis is valid, we need models that converge monotonically
and deterministically in solo mode across 10 epochs.** The current synthetic
datasets and/or LR settings are inadequate for some models.

## Weighted AllReduce: Implementation Notes

### Stream ordering bug
`mul_scalar_(weight)` enqueues on the default/compute stream. `all_reduce_on_stream`
enqueues on comm_stream. Without synchronization, AllReduce starts before scaling
completes, summing unscaled params. Params double each sync -> exponential blowup.

**Fix:** `cuda_synchronize(device.index())` before AllReduce. Heavy hammer but correct.
StreamGuard alone was NOT sufficient (libtorch tensor ops may not respect the "current
stream" set by StreamGuard in all cases).

### Weight ratio is NOT 19x
Debug output showed per-sync weights of ~0.44/0.54, not the expected 0.94/0.06.
Cadence triggers on wall-time (anchor_wall_ms), so both GPUs accumulate similar
wall time between syncs. But the fast GPU should do ~19x more BATCHES in that time.
This discrepancy needs investigation -- either ElChe is capping the batch ratio,
or the steps_since_avg counting has an issue.

### The cuda_synchronize cost
Adds a full device sync before every AllReduce. The existing comment says
"No throughput loss: compute_stream already waits on copy_done, so AllReduce
and compute can't overlap." But the sync stalls the GPU pipeline, which may
explain why MLP got worse (the overhead accumulates over 266 syncs).

## Key Insights

### 1. Solo baselines must be fixed first
MLP diverges on solo-0 at 10 epochs. Linear plateaus at a high floor. Without
reliable baselines, we cannot measure DDP convergence quality.

**Action:** Fix all 9 models to converge monotonically in solo mode at 10 epochs.
This may require: different LR schedules, LR warmup, dataset changes, or
removing the auto-scale (0.50x) that halves LR for 10 vs 5 epochs.

### 2. Epoch boundary as convergence guardrail
User hypothesis: epoch boundaries force a hard rendez-vous where both GPUs finish
their partition before the next epoch starts. This creates a natural convergence
checkpoint. However, an AllReduce at the epoch boundary is mathematically identical
to any other AllReduce -- the only special thing is LR scheduler steps (if any).
The current bench uses flat LR, so epoch position shouldn't matter.

### 3. Cadence AllReduce disrupts optimizer state
Each AllReduce(Avg) perturbs Adam's momentum/variance estimates. With ~56 batches
between syncs (179 syncs / 10k batches), the optimizer may not fully recover before
the next sync. This "perpetual recovery" could explain oscillation patterns.

### 4. Weighted averaging: marginal at best, harmful at worst
Per-sync weight ratio is ~0.46/0.54 (not 19x). The mathematical correction is tiny.
The implementation cost (cuda_synchronize, two ops instead of one atomic AllReduce)
outweighs the benefit. MLP got 2.7x worse with weighting.

### 5. DDP can IMPROVE convergence
MLP cadence (0.037) vastly outperforms MLP solo (0.511). The parameter averaging
acts as regularization, preventing the single-GPU optimizer from diverging.
This is well-documented in Local SGD literature.
